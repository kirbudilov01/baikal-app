# Байкал в наших руках: мастер-план готового решения

## 1. Главный смысл продукта

Приложение должно быть не просто формой жалобы. Его смысл: превратить неравнодушных жителей и гостей Иркутской области в распределенную сеть раннего обнаружения экологических и инфраструктурных проблем, а затем показать человеку, что его сообщение не пропало в пустоту.

Главная эмоция пользователя: "Я заметил проблему, быстро сообщил, вижу статус, получил подтверждение, что мой голос что-то меняет".

Главная ценность для организации: поток структурированных заявок с фото, геоточкой, категорией, статусом проверки, историей действий и понятным маршрутом передачи ответственным службам.

Главная ценность для партнеров: бонусная программа за полезные действия, которая связывает экологическую активность с локальными сервисами Байкала.

## 2. Ключевые принципы

1. Минимум трения при создании сообщения.
   Пользователь должен отправить проблему за 30-60 секунд: фото, категория, точка, короткое описание.

2. Не обещать больше, чем можем выполнить.
   Статусы должны отражать реальный процесс: получено, проверяется, передано, в работе, решено, отклонено с причиной.

3. Анонимность и доверие.
   Пользователь может оставаться анонимным для внешних служб. При этом система должна уметь защищаться от спама и фейковых заявок.

4. Прозрачность.
   Каждая заявка имеет историю: кто и когда изменил статус, какой следующий шаг, есть ли официальный ответ.

5. Проверяемость.
   Фото, геопозиция, дубликаты, подтверждения других пользователей и модерация должны помогать отличать реальную проблему от шума.

6. Геймификация без манипуляции.
   Баллы выдаются за полезные действия, но не за накрутку. Баллы не должны подталкивать к ложным сообщениям.

7. Модерация с первого дня.
   Любой пользовательский контент: фото, текст, комментарии, подтверждения. Значит, нужна система жалоб, скрытия, удаления, блокировок и правил.

## 3. Основные аудитории

### Житель региона

Задачи:
- быстро сообщить о мусоре, вырубке, свалке, загрязнении, опасном месте;
- понять, что заявка принята;
- получать уведомления о прогрессе;
- зарабатывать баллы за полезные действия.

Риски:
- не доверяет, что сообщение дойдет;
- боится раскрывать личность;
- не хочет заполнять длинную форму.

### Турист

Задачи:
- сообщить о проблеме рядом с маршрутом;
- не разбираться в местной бюрократии;
- получить простое подтверждение и, возможно, бонус у партнеров.

Риски:
- не знает точное название места;
- плохой интернет;
- не хочет регистрироваться.

### Модератор

Задачи:
- быстро отсеять спам, дубликаты, невалидные заявки;
- уточнить категорию и место;
- объединять похожие сообщения;
- передавать подтвержденные проблемы дальше.

Риски:
- поток заявок станет неуправляемым;
- много повторов;
- трудно доказать реальность проблемы.

### Координатор/администратор проекта

Задачи:
- видеть общую картину по региону;
- контролировать SLA;
- готовить отчеты;
- отслеживать эффективность служб и партнеров;
- управлять категориями, баллами, партнерами, пользователями.

### Ответственная служба или партнерская организация

Задачи:
- получать только релевантные подтвержденные заявки;
- давать статус или официальный ответ;
- видеть материалы: фото, координаты, описание, историю.

### Партнер бонусной программы

Задачи:
- размещать предложение;
- принимать баллы/промокоды;
- видеть статистику активаций;
- не получать мошеннические списания.

## 5. Полная карта функций

### 5.1. Онбординг

Нужно:
- первый экран с краткой миссией;
- выбор режима: продолжить анонимно / войти по телефону / войти через Apple/Google;
- принятие пользовательского соглашения и политики обработки данных;
- запрос разрешений постепенно, а не сразу:
  - геолокация при выборе точки;
  - камера при добавлении фото;
  - пуши после первой успешной заявки;
- выбор района/интересующей территории;
- объяснение баллов без обещаний денежных выплат.

Результат:
- пользователь понимает, зачем приложение;
- юридические согласия собраны;
- разрешения запрошены в правильный момент.

### 5.2. Авторизация и аккаунт

Режимы:
- анонимный пользователь с локальным device id;
- подтвержденный аккаунт по телефону;
- вход через Apple;
- вход через Google;
- служебные аккаунты для модераторов и администраторов.

Поля пользователя:
- id;
- display name;
- phone/email, если есть;
- auth provider;
- avatar, опционально;
- регион;
- уровень доверия;
- баланс баллов;
- флаги блокировки;
- дата регистрации;
- дата последней активности;
- согласия с документами;
- настройки уведомлений.

Важно:
- анонимный режим должен позволять отправлять заявку, но с лимитами;
- восстановление аккаунта нужно для баллов;
- удаление аккаунта и выгрузка данных обязательны для нормального релиза.

### 5.3. Создание сообщения о проблеме

Минимальный сценарий:
1. Пользователь нажимает "Сообщить".
2. Делает фото или выбирает из галереи.
3. Выбирает категорию.
4. Ставит точку на карте или подтверждает текущую геопозицию.
5. Пишет короткое описание.
6. Отправляет.
7. Получает экран успеха и начисление предварительных баллов.

Расширенный сценарий:
- несколько фото;
- видео до короткого лимита;
- офлайн-черновик;
- автоподстановка адреса по координатам;
- предупреждение о похожих заявках рядом;
- возможность подтвердить существующую заявку вместо создания дубля;
- выбор срочности;
- чекбоксы "опасно для людей", "нужна срочная уборка", "есть техника/люди на месте";
- скрытие точных персональных данных перед отправкой.

Валидация:
- фото обязательно для части категорий;
- координаты обязательны;
- описание минимум 10-20 символов;
- защита от повторной отправки;
- лимиты на число заявок в день для анонимов;
- проверка подозрительных фото и текста.

### 5.4. Категории проблем

Базовые:
- незаконная вырубка;
- мусор;
- стихийная свалка;
- загрязнение воды;
- строительство без разрешения;
- разлив нефтепродуктов;
- повреждение природы;
- другое.

Расширить:
- кострища/пожары;
- браконьерство;
- незаконный проезд транспорта;
- повреждение туристической инфраструктуры;
- опасные отходы;
- нарушение водоохранной зоны;
- шум/незаконные работы;
- животные в опасности;
- vandalism/порча знаков и троп.

Для каждой категории задать:
- обязательные поля;
- подсказку пользователю;
- приоритет по умолчанию;
- кому передается;
- начисление баллов;
- правила проверки;
- шаблон обращения в службу;
- SLA.

### 5.5. Карта

Функции:
- карта всех публичных подтвержденных проблем;
- фильтры по категории, статусу, периоду;
- кластеризация;
- карточка заявки на карте;
- "подтвердить проблему";
- "проблема уже решена";
- скрытие точных координат для чувствительных категорий;
- тепловая карта для админов;
- офлайн-кэш базовой карты в ключевых местах.

Интеграции:
- Yandex Maps или 2GIS для РФ-сценария;
- возможно Mapbox/OpenStreetMap, если нужна независимость;
- reverse geocoding;
- хранение геометрии в PostGIS.

### 5.6. Мои сообщения и статусы

Статусы заявки:
- черновик;
- отправлено;
- получено системой;
- на модерации;
- нужна дополнительная информация;
- подтверждено;
- объединено с другой заявкой;
- передано ответственной службе;
- принято в работу;
- ожидаем официальный ответ;
- решено;
- отклонено;
- архив.

Для пользователя показывать упрощенно:
- получено;
- проверяется;
- передано;
- в работе;
- решено;
- требуется уточнение;
- отклонено с причиной.

Для админки хранить детально:
- кто изменил статус;
- дата;
- комментарий;
- вложения;
- внешний номер обращения;
- SLA deadline;
- next action.

### 5.7. Подтверждения другими пользователями

Функции:
- пользователь рядом может подтвердить проблему;
- можно добавить новое фото;
- можно отметить "уже решено";
- подтверждения повышают доверие заявки;
- подозрительные массовые подтверждения не учитываются.

Антифрод:
- учитывать расстояние до точки;
- учитывать возраст аккаунта;
- device fingerprint;
- лимиты;
- вес голоса по уровню доверия;
- поиск групповой накрутки.

### 5.8. Баллы и достижения

События начисления:
- заявка с фото: +50;
- подтверждение другой заявки: +20;
- заявка прошла модерацию: +30;
- проблема решена: +100;
- активность в сообществе: +10;
- серия полезных действий: бонус;
- ложная заявка: 0 и возможный штраф доверия.

Правила:
- часть баллов может быть "ожидает подтверждения";
- баллы становятся доступными после модерации;
- за дубликаты баллы не начисляются или начисляются меньше;
- за вредоносные действия баланс не уходит в минус, но падает trust score;
- история начислений обязательна.

Достижения:
- "Первое сообщение";
- "Защитник Байкала";
- "Помощник модерации";
- "Чистый берег";
- "10 подтвержденных проблем";
- сезонные кампании.

### 5.9. Бонусы и партнеры

Функции:
- каталог предложений;
- фильтры: еда, жилье, экскурсии, прокат, сувениры;
- стоимость в баллах;
- условия;
- срок действия;
- генерация промокода/QR;
- списание баллов;
- проверка партнером;
- история использований.

Сущности:
- partner;
- offer;
- redemption;
- promo code;
- partner staff account.

Антифрод:
- одноразовые коды;
- срок действия кода;
- подтверждение партнером;
- отмена списания при ошибке;
- лимиты активаций.

### 5.10. Уведомления

Push-события:
- заявка получена;
- заявка прошла модерацию;
- требуется уточнение;
- заявка передана;
- статус изменен;
- проблема решена;
- начислены баллы;
- рядом есть проблема для подтверждения;
- появился новый бонус.

Каналы:
- push;
- email для зарегистрированных;
- Telegram-уведомления как возможная интеграция;
- in-app inbox.

Настройки:
- статусы моих заявок;
- баллы;
- проблемы рядом;
- новости проекта;
- партнерские предложения.

## 6. Админка

Админка нужна почти сразу. Без нее приложение будет красивой формой, но не системой.

### 6.1. Роли

Роли:
- super admin;
- project admin;
- moderator;
- regional coordinator;
- service partner;
- reward partner;
- analyst;
- read-only observer.

Права:
- просмотр заявок;
- изменение статуса;
- назначение ответственного;
- объединение дубликатов;
- скрытие контента;
- блокировка пользователя;
- управление категориями;
- управление баллами;
- управление партнерами;
- экспорт отчетов;
- доступ к персональным данным.

### 6.2. Очередь модерации

Нужно:
- список новых заявок;
- фильтры по категории, району, срочности, подозрительности;
- быстрый просмотр фото;
- карта рядом;
- похожие заявки;
- кнопки:
  - подтвердить;
  - запросить уточнение;
  - объединить;
  - отклонить;
  - скрыть;
  - передать;
- шаблоны комментариев.

### 6.3. Карточка заявки в админке

Поля:
- фото/видео;
- категория;
- описание;
- координаты;
- адрес;
- автор;
- trust score автора;
- похожие заявки;
- подтверждения;
- история статусов;
- внутренние комментарии;
- публичные комментарии;
- документы/ответы;
- внешний номер обращения;
- назначенный ответственный;
- SLA.

### 6.4. Передача ответственным службам

Минимальный вариант:
- email-шаблон с материалами;
- экспорт PDF;
- отметка "передано";
- ручное внесение ответа.

Продвинутый вариант:
- интеграция с CRM/СЭД;
- API для служб;
- личный кабинет службы;
- статусы от службы;
- вложения и официальные ответы;
- контроль сроков.

### 6.5. Аналитика

Дашборды:
- количество заявок по категориям;
- карта проблем;
- среднее время модерации;
- среднее время решения;
- SLA overdue;
- активные пользователи;
- конверсия: создано -> подтверждено -> передано -> решено;
- самые проблемные районы;
- эффективность партнеров;
- стоимость бонусной программы;
- подозрительная активность.

## 23. Главные риски

1. Нет реального процесса обработки.
   Если заявки никто не берет в работу, доверие рухнет.

2. Много спама и дублей.
   Нужны лимиты, модерация, дубликаты, trust score.

3. Юридическая неготовность.
   Фото, геолокация и пользовательский контент требуют документов и процедур.

4. Непрозрачные баллы.
   Если правила начисления неясны, будет недоверие и накрутка.

5. Слишком широкий запуск.
   Лучше пилот на ограниченной территории, чем сразу весь регион без процесса.

6. Нет партнерской операционки.
   Бонусы требуют учета, проверки кодов и поддержки.

## 26. Определение "готово"

Продукт можно считать готовым к пилоту, когда:
- пользователь может отправить реальную заявку с фото и геоточкой;
- модератор видит заявку в админке;
- модератор меняет статус;
- пользователь получает статус и пуш;
- заявка хранит историю;
- есть правила UGC и жалобы;
- есть базовые документы;
- есть crash/error monitoring;
- есть минимум один сценарий бонусов или он отключен feature flag;
- команда умеет обработать входящий поток.

Продукт можно считать готовым к публичному релизу, когда:
- пройден пилот;
- есть рабочий процесс ответственных;
- закрыты юридические документы;
- работает удаление аккаунта;
- работает модерация и блокировки;
- заполнены store privacy/data safety формы;
- есть production backend;
- есть резервное копирование;
- есть поддержка пользователей;
- есть план реакции на инциденты.
